# 视频播放器音画同步与管线崩溃终极解决方案

本文档记录了在 VideoDecoder 项目中，解决“音画不同步”、“黑屏死循环”、“视频光速快进”、“只有一帧画面”等一系列恶性播放 Bug 的完整排查过程与最终解决方案。

## 1. 现象回顾

在开发和调试基于 FFmpeg + OpenGL ES + AAudio 的 Android C++ 播放器时，我们经历了以下几个阶段的致命问题：
1. **黑屏且视频光速结束**：画面一直黑屏，直到音频快播完时突然闪现最后一帧然后结束。
2. **视频永远慢于音频**：不管怎么调参数，画面永远追不上声音。
3. **视频快进且无声**：画面像快进一样瞬间播完，同时完全没有声音。
4. **疯狂卡顿**：真机运行虽然同步了，但是画面像抽帧的幻灯片一样卡顿。

## 2. 核心问题定位与深度修复

经过对底层 C++ 源码的逐步解剖，我们发现问题并非单一的“时间戳算错”，而是**多线程调度、时钟基准、以及生产者消费者模型**三个维度的逻辑缺陷叠加导致的。最终的解决方案涵盖了以下四大核心修复：

### 2.1 修复“生产者失控”与“满载丢帧” (解决黑屏/只有一帧)
**【致命缺陷】**：
原先的 `FrameQueue` 和 `PacketQueue` 没有正确实现阻塞机制。尤其是 `FrameQueue::push`，当队列满了时，它的逻辑居然是**“强行弹出并丢弃最老的帧”**来腾出空间。
由于视频渲染线程需要 `Sleep` 来等待音频同步，消费速度会降下来；而解码线程在后台全速运行。这导致解码线程以光速解完了整个视频，并把新帧全塞进队列，把本来该渲染的旧画面全扔进了垃圾桶。最终用户只能看到最后一帧。

**【最终方案】**：
严格落实**生产者-消费者阻塞模型**。在 `queue.cpp` 和 `queue.h` 中，利用 `std::condition_variable` 的 `wait` 方法。
* 当 `PacketQueue` 达到 100 时，Demuxer 线程必须阻塞等待。
* 当 `FrameQueue` 达到 10 时，解码线程必须阻塞等待，**绝对禁止抛弃未渲染的旧画面**。

### 2.2 剥除“双重休眠”枷锁 (解决视频永远慢于音频)
**【致命缺陷】**：
在 `native-lib.cpp` 的渲染循环中，除了用于音画同步的 `sleep_for(diff)` 之外，紧接着还有一段老旧的“帧率控制逻辑”（如果渲染耗时小于 `frameDuration` 如 33ms，则强制休眠补齐 33ms）。
这意味着，即便视频已经落后于音频需要全速追赶，它依然被强制锁死在 30 帧的上限之下，加上软解耗时，导致视频永远追不上声音。

**【最终方案】**：
完全**移除基于帧率的双重 Sleep 控制**。只要系统中存在音频（`g_audioClock > 0`），视频的播放节奏就**完全交由音频时钟来指挥**。视频快了就等，视频慢了就全速渲染，彻底解开性能枷锁。

### 2.3 确立“绝对精准的硬件时钟” (解决同步偏差)
**【致命缺陷】**：
仅仅使用 `g_audioClock`（刚塞进队列的音频 PTS）作为当前时间戳是完全错误的，因为 AAudio 底层硬件和我们的代码队列中都积压了大量还没播出的数据。如果混用 `Float` 和 `Int16` 字节大小去计算延迟，会导致计算出的微秒延迟全错。

**【最终方案】**：
引入了最精确的 `exact_audio_pts` 计算公式：
1. **确立队列延迟**：通过 `queuedAudioBytes / (channels * sizeof(int16_t))` 算出严格的 S16 格式积压延迟。
2. **确立硬件延迟**：利用 `AAudioStream_getFramesWritten - getFramesRead` 算出硬件层还没发声的帧数。
3. **真实时间戳**：`当前正在发声的时间 = 最新解码的 PTS - 队列延迟 - 硬件延迟`。

### 2.4 柔性双向对齐与防死锁保护 (解决抽风卡顿)
**【致命缺陷】**：
早期的同步策略极其暴躁。一旦落后超过 40ms，就强制 `av_frame_free` 丢帧。由于虚拟机或老旧手机纯软解性能较差，必然会导致频繁丢帧，引发“幻灯片”或者“死循环黑屏”。如果视频太快，又直接 `sleep(1000ms)` 导致画面长时间冻结。

**【最终方案】**：
实施了**柔性对齐策略**：
* **视频太快时（`diff > 0`）**：执行 Sleep，但设置单次最大休眠上限为 `100ms`（防止首帧 PTS 异常引发长达数秒的死锁黑屏）。
* **视频太慢时（`diff < 0`）**：**彻底废除强制丢帧逻辑**。只要解码器性能跟得上，就让它跳过 Sleep 以 CPU 最大算力全速渲染。虽然极少数情况下可能轻微晚于音频，但这保证了画面的绝对流畅连贯，符合人类在观看视频时“对画面卡顿极其敏感，对极其微小声音落后相对宽容”的视觉心理学。

## 3. 最终同步算法伪代码速览

```cpp
// 1. 获取最精确的音频发声时间
int64_t exact_audio_pts = g_audioClock.load() - g_audioRenderer->getPendingAudioDurationUs();
int64_t video_pts = g_videoClock.load();

if (exact_audio_pts > 0 && video_pts > 0) {
    int64_t diff = video_pts - exact_audio_pts;

    if (diff > 0) {  
        // 视频比音频快，必须休眠等待音频
        int64_t sleep_time = diff;
        // 防死锁护城河：单次最多只睡 100ms
        if (sleep_time > 100000) sleep_time = 100000; 
        std::this_thread::sleep_for(std::chrono::microseconds(sleep_time));
    }
    // diff <= 0 时，视频落后，不睡、不丢帧，全速渲染追赶
}

// 执行渲染
renderer.renderFrame(frame);
eglSwapBuffers(display, surface);
```

## 4. 总结

Android C++ 原生播放器的音画同步是一个极其精密的**多线程 PID 控制系统**。
绝不能孤立地看待时间戳，必须将 **“队列管控（防 OOM/防丢老帧）”**、**“声卡硬件延迟折算”** 以及 **“人眼视觉平滑容忍度”** 结合在一起。本次重构彻底打通了三大护城河，让 VideoDecoder 蜕变为了一个稳定、流畅且具有商业级防崩溃能力的 Native 播放器内核。
